----------------------------------------------------------------
RÉSUMÉ DU PROJET
----------------------------------------------------------------

Objectif :
  Prédire si la condition de la valve d'un système hydraulique
  industriel est optimale (1) ou non (0) à partir de données
  capteurs, et expliquer les causes de dégradation via SHAP.

Dataset :
  UCI Condition Monitoring of Hydraulic Systems
  - 2205 cycles de production
  - PS2 : capteur de pression (100 Hz, 6000 points/cycle)
  - FS1 : capteur de débit (10 Hz, 600 points/cycle)
  - profile.txt : labels (valve_condition : 100, 90, 80, 73)

Cible :
  - 1 = Optimal       (valve_condition == 100)
  - 0 = Non-optimal   (valve_condition < 100)

Split train/test :
  - Train : cycles 0 à 1999    (2000 cycles)
  - Test  : cycles 2000 à 2204 (205 cycles) — jamais touché avant l'évaluation finale

----------------------------------------------------------------
QUESTIONS PROBABLES DU JURY
----------------------------------------------------------------

Q : Pourquoi le score est-il de 1.0 ? C'est du overfitting ?
R : Non. Le test set (cycles 2000-2204) n'a jamais été vu pendant
    l'entraînement ni le tuning. Le score de 1.0 reflète la qualité
    du signal industriel : les capteurs de pression (PS2) produisent
    des signaux très distincts selon l'état de la valve.
    Ce résultat est cohérent avec la littérature académique sur ce dataset UCI.

Q : Pourquoi PS2 et FS1 séparément ?
R : Les fréquences d'échantillonnage sont différentes (100 Hz vs 10 Hz).
    On ne peut pas concaténer des tableaux de tailles différentes.
    On agrège d'abord chaque capteur par cycle (stats + FFT), puis on combine.

Q : Pourquoi le Random Forest plutôt que LightGBM ou XGBoost ?
R : RF offre le meilleur compromis F1/vitesse sur ce dataset.
    Il est aussi plus interprétable via SHAP (TreeExplainer natif).
    LightGBM est légèrement moins performant ici et plus lent à tuner.

Q : C'est quoi Optuna ?
R : Un framework d'optimisation automatique des hyperparamètres.
    Il explore intelligemment l'espace (algorithme TPE) pour trouver
    les meilleurs n_estimators, max_depth, etc. sans grid search exhaustif.

Q : Pourquoi F1 comme métrique principale ?
R : Parce qu'il y a un déséquilibre de classes (plus de cycles non-optimaux
    que optimaux). F1 pénalise les faux positifs ET les faux négatifs,
    ce qui est adapté à un contexte industriel où une valve dégradée
    non détectée peut causer une panne coûteuse.

Q : Comment fonctionne l'API ?
R : On envoie un POST /predict avec un cycle_id.
    L'API retourne : prediction (0 ou 1) + probability_optimal.
    Les données PS2/FS1 sont chargées en mémoire au démarrage (lifespan)
    pour éviter de relire 78 MB à chaque requête.

Q : Qu'est-ce que SHAP apporte ?
R : SHAP explique POURQUOI le modèle prédit non-optimal pour un cycle donné.
    Il identifie que les features PS2 (pression) sont les plus impactantes.
    C'est crucial en industrie : l'opérateur sait sur quel capteur agir.

Q : Qu'est-ce que Prometheus/Grafana surveille ?
R : Latence des requêtes (p50/p95/p99), taux d'erreur HTTP,
    nombre de requêtes par seconde, consommation CPU et mémoire de l'API.

Q : C'est quoi le CI/CD ?
R : À chaque push sur GitHub : lint (ruff + black) + tests unitaires
    + build Docker automatique. Si un test échoue, le push est bloqué.

Q : Pourquoi DVC ?
R : Les fichiers de données (PS2.txt, FS1.txt) sont trop lourds pour git.
    DVC les versionne séparément (comme git pour le code) sur un stockage externe.
